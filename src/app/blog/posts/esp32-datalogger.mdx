> **Status:** Functional Prototype  
> **Role:** Hardware Design, Firmware Development, Networking, Data Processing  
> **Technologies:** ESP32, C++, Arduino IDE, Python, TCP/IP Networking, MPU6050, BMP280

---

## Project Summary

This project was originally developed as a lightweight flight datalogger for an RC aircraft being built by a friend. The objective was to collect motion and environmental data during flight and store it for post-flight analysis.

The system uses an ESP32-WROVER-E microcontroller to collect data from an MPU6050 inertial measurement unit and a BMP280 environmental sensor. Sensor readings are processed onboard, formatted into telemetry packets, and transmitted wirelessly over WiFi to a laptop. A custom Python application receives the incoming data stream and automatically saves the results to timestamped CSV files for later analysis.

Although the RC aircraft project was ultimately discontinued, the datalogger itself was successfully developed into a working prototype and demonstrated the complete data acquisition pipeline from sensor collection through data storage.

The system consists of three primary components:

1. **Embedded Sensors** Collects the raw data
2. **Microcontroller** Calculating, formats, and sends desired data

## Firmware Architecture

The ESP32 firmware executes the following sequence:

![ESP32 Datalogger Firmware Architecture](/images/projects/01_Datalogger/FirmwareArchitecture.png)

*Figure 2. High-level Firmware architecture showing I2C connections, ESP32 setup and continuous behavior, and python script flow.*


**Sensor Acquisition**

Sensor communication is performed over I2C with each sensor having a unique hardware address.

The firmware continuously gathers from:

- **MPU6050:** Accelerometer, Gyroscope data
- **BMP280:** Pressure, Altitute, Temperature data

**Orientation Calculation**

Roll, pitch, and yaw are calculated using accelerometer measurements and gyroscope integration.

The `calculate_IMU_error()` function calculates the error in gyro and acceleration measurements due to drift. This value is used as an offset for the continually calculated values to minimize measurement drift. 

**Telemetry Formatting**

Sensor data is assembled into structured packets before transmission to allow simple parsing on the receiving system.

Example packet:

```text
packetNumber,rssi,roll,pitch,yaw,altitude,pressure,temperature
```

## Problems Faced

**IMU Calibration**

Gyroscopes accumulate error over time when angular velocity is integrated into position estimates. Raw sensor measurements also contain bias and noise.

To improve accuracy, average drift values are calculated at startup (while stationary), and used as offset values to improve long term orientation accuracy.

---

**Wireless Communication Reliability**

Depending on the strength of the WiFi source, the connection can eventually drop between devices. WiFi strength (RSSI) was transmitted with every data package so users can determine viable operating ranges.
