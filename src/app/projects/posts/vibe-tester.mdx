## Project Summary

The objective of this project was to develop a low-cost vibration testing system for RIT Racing to evaluate the durability of custom PCBs used in their Formula Student vehicle. The system simulates the high-frequency vibrations and mechanical stresses experienced during high-speed driving and aggressive cornering in competition.

This was a group project, with my contributions focused on the entire electronics stack, including firmware development, hardware design, soldering, and system integration. The system uses an ESP32 to generate programmable sine-wave signals through its DAC, which are conditioned and used to drive a speaker-based vibration platform. A built-in LCD and keypad interface enables fully standalone configuration and automated frequency sweep testing.

## System Architecture

The vibration tester consists of three primary subsystems:

1. **UI Peripherals**: Determine what signal needs to be produced
2. **Microcontroller**: Produces the signal + manages the UI 
3. **Signal Chain**: Converts the signal to physical vibration

![ESP32 Vibe Tester System Architecture](/images/projects/02_VibeTester/SystemArchitecture.jpg)

*Figure 1. High-level system architecture showing input peripherals, ESP32 core tasks, and output signal chain*

**System Operation**

1. User enters testing parameters through the keypad.
2. ESP32 processes input and updates configuration settings.
3. Firmware generates a sine wave using the onboard DAC.
4. Signal conditioning circuitry filters and adjusts the output signal.
5. Speaker converts the electrical signal into mechanical vibration.
6. PCB mounted to the test platform experiences controlled vibration.
7. Frequency is automatically swept between configured limits.

## Firmware Architecture

The firmware is responsible for user interaction, parameter storage, signal generation, and overall system control.

![Firmware Architecture](/images/projects/02_VibeTester/FirmwareArchitecture.png)

*Figure 2. High-level firmware architecture showing menu navigation, parameter configuration, and vibration control logic.*

**Core 0**
1. Refreshes the screen whenever the running state has changed
2. Listens for key presses and navigates the menu accordingly
3. Handles user input for configuring test parameters
4. Starts and stops the vibration test based on user input

**Core 1**
1. Sits idle until the test is started
2. Drives the speaker at the current frequency for a set duration
3. Steps to the next frequency, bouncing back when the upper or lower limit is reached
4. Signals Core 0 to update the display after each frequency change

**Frequency Sweep Generation**

The vibration profile is generated by producing a sine wave through the ESP32 DAC.

The waveform follows:

```
V(t) = AMPLITUDE * sin(2π * frequency * t) + VOLT_OFFSET
```

```t``` is the time in seconds with accuracy to the microsecond. 

The frequency of the sine wave varies between the minimum and maximum frequency values provided in the configuration settings. The size of the ```step``` (Hz) between frequencies and the ```timePerFrequency``` (ms) are also provided in the configuration settings.

## Problems Faced

**Balancing Signal Generation and User Interface Updates**

Initially, all processes were executed in the main loop. Because of this, everytime the UI was updated to reflect the current status of the test, signal generation was delayed temporarily.

This problem was solved implementing task seperation through multi-core programming. The main processing of each tasks was seperated to its own dedicated core, therefore making the overall execution of the firmware much more consistent.  

---

**DAC Resolution Limitations**

The ESP32's onboard DAC provides limited 8-bit output resolution.

The allowed voltage supplied to the speaker was much lower than the capacity of the ESP32, so  initially I had the min and max voltages set lower in the code so it could be directly fed to the speaker. 

This introduced resolution issues, because of the 8-bit accuracy of the ESP32's DAC, by lowering the voltage range, I also unintentionally lowered the possible resolution of the waveform. 

This problem was solved by using the full range of the ESP32's DAC (0-3.3V), and then stepping down + filtering that voltage using an external voltage divider and filter so that the speaker would receive safe voltage levels without degrading signal resolution.

---

**Hardware Integration**

There was a lot debugging and design changes throughout this project since it was the first time I soldered, and lots of mistakes were made.

- Using unreliable/weak wires led to many wires breaking resulting in de/resoldering
- Cable management inside the electronics housing was initially very messy, unreliable, and bad for longevity.
- An LCD screen's brightness ability broke entirely due to not being properly taken care of

With time and the guidance/advice of other senior members of the electronics team, I was able to resolve all of these problems, and produce a reliable product, and learn a lot along the way.
